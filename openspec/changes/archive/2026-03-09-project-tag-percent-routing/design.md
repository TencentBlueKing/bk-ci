## Context

当前项目路由切换依赖运维接口 `setTagByProject` 按 `projectId` 单条执行，适合小规模应急处理，但不适合灰度发布期间的大规模放量。
现网存在两套目标集群：灰度 `rbac-gray` 与生产 `rbac`。发布后希望按比例逐步将项目路由迁移到灰度集群，并在验证通过后推进到全量。

约束与关注点：
- 必须兼容现有按项目切换接口，避免影响当前运维链路。
- 放量过程需要可回退、可审计、可重复执行（幂等）。
- 项目选择必须稳定可复现，避免同一比例多次执行命中集合波动过大。
- 部分项目需要强制包含（白名单）或强制排除（黑名单），不受哈希比例影响。

## Goals / Non-Goals

**Goals:**
- 提供"按比例切换项目路由 tag"的运维能力，支持 1%/3%/5%/10%/.../100% 的阶段化推进。
- 提供确定性的项目抽样策略，使每个比例对应的项目集合可复现。
- 支持强制路由白名单，确保指定项目一定被切换到目标 tag。
- 支持路由排除名单（黑名单），确保指定项目一定不参与本次放量。
- 提供执行结果统计与失败明细，支持快速回滚与补偿重试。
- 保持与现有单项目接口并存，支持混合运维策略。

**Non-Goals:**
- 不改动业务方调用 Project 接口的方式（仅运维侧能力增强）。
- 不在本变更中引入新的流量网关或请求实时分流组件。
- 不覆盖跨模块的自动发布编排（由外部发布系统决定何时调用）。

## Decisions

### 决策 1：新增"按比例放量"运维接口，而非改造单项目接口语义
- 方案：新增独立接口用于比例放量，输入包含目标比例、源/目标 tag、可选过滤条件与执行模式（预览/执行）。
- 原因：单项目接口语义简单且已被现有工具依赖，直接改造风险高。
- 备选：
  - 复用单接口并扩展参数：会使接口语义混杂，误调用风险高。
  - 完全替换旧接口：迁移成本高，回滚路径复杂。

### 决策 2：使用确定性哈希分桶进行项目抽样
- 方案：对 `projectId` 做稳定哈希（如 Murmur/CRC32），映射到 `[0, 99]` 桶，比例阈值决定目标集合。
- 原因：同一比例多次执行结果一致，可预测、可审计，且无需额外持久化抽样名单。
- 备选：
  - 每次随机抽样：实现简单但不可复现，不利于回滚和问题排查。
  - 持久化抽样名单：一致性强，但存储和维护复杂度更高。

### 决策 3：接口支持 dry-run 与 execute 两种模式
- 方案：`dry-run` 返回将要切换的项目统计和样本，`execute` 真正落库并返回结果。
- 原因：放量前可先评估影响范围，降低误操作风险。
- 备选：
  - 只提供 execute：操作快，但风险大，问题发现滞后。

### 决策 4：执行过程要求幂等与增量推进
- 方案：当目标比例从 3% 推到 5% 时，仅对新增命中的项目执行切换；已在目标 tag 的项目跳过并计入已达成。
- 原因：减少重复写入与变更噪声，便于统计增量效果。
- 备选：
  - 每次覆盖全量命中集合：实现简单但写放大明显，审计信息噪声高。

### 决策 5：白名单与排除名单持久化到 Redis，全局唯一，不按 tag 区分
- 方案：白名单与黑名单分别存储为固定 Redis Set（`project:routing:whitelist` /
  `project:routing:blacklist`），与 targetTag 无关。
  提供独立的增删查接口管理名单内容，放量接口执行时从 Redis 读取当前名单。
- 优先级（由高到低）：排除名单 > 强制路由白名单 > 哈希百分比选择。
  即同时出现在两个名单中的项目，以排除名单为准，不会被切换。
- 原因：
  - 白名单/黑名单是针对特定项目本身的固定属性（如"VIP项目永远优先灰度"或"维护中项目永不路由"），
    与当前发布的目标 tag 无关，不应随 tag 变化而变化。
  - 全局名单更易于运维管理，避免每次发布新灰度版本都重新维护同一份名单。
  - Redis Set 支持 O(1) 成员判断，大名单下性能优于遍历参数列表。
- 备选：
  - 按 targetTag 区分：灵活性更高，但增加维护成本，且实际场景中名单内容不随 tag 变化。
  - 接口参数传入：无需额外存储，但每次放量都需携带完整名单，跨阶段一致性由调用方保证，易出错。

### 决策 6：哈希采用固定范围 `[0, 99]`，与项目总数无关
- 方案：`hash(projectId) % 100`，阈值 = `targetPercent`，命中条件为哈希值 < 阈值。
- 原因：新增项目不影响已有项目的哈希结果，集合稳定可复现。
  如改用 `% N`（N=总数），每次项目新增都会导致映射漂移，破坏集合稳定性。
- 备选：
  - `hash(projectId) % N`：实现简单但集合不稳定，不可采用。

### 决策 7：`channelCode` 保持字符串并提供默认值
- 方案：`ProjectPercentageRoutingRequest.channelCode` 保持字符串类型；
  当请求未传该字段时，服务端按默认值 `BS` 处理。
- 原因：
  - 与现有 `ProjectConditionDTO.channelCode` 参数类型保持一致，避免额外类型转换与跨模块改造。
  - 兼容历史调用方，降低升级成本。
- 备选：
  - 使用 `ChannelCode` 枚举：类型更强，但会引入依赖与兼容改造成本，本次不采用。

## Risks / Trade-offs

- [风险] 哈希分桶与历史人工切换结果不一致，导致阶段统计出现"已达成"偏差
  Mitigation：返回"已在目标 tag"与"本次实际切换"分项统计，并在文档中明确口径。
- [风险] 大比例切换时批量更新带来数据库写压力
  Mitigation：分批执行（batch）、限流与重试；失败项收集后异步补偿。
- [风险] 回滚时需要快速将目标集合恢复到 `rbac`
  Mitigation：提供反向执行参数（source/target 可互换）并复用同一选择算法。
- [取舍] 选择确定性哈希牺牲"完全随机公平"
  Mitigation：通过统一哈希算法与盐值管理保证长期分布稳定。
- [风险] Redis 名单数据意外丢失（如 Redis 重启、key 过期）会导致放量时名单为空，白名单项目被遗漏
  Mitigation：Redis key 不设过期时间；名单管理接口提供查询能力，放量前可先 dry-run 确认名单生效；
  必要时可在放量接口中增加"名单为空时是否继续"的安全开关。
- [风险] 多环境（测试/预发/生产）共用 Redis 时名单可能互相污染
  Mitigation：通过 Redis 命名空间或不同 Redis 实例区分环境，名单 key 本身无需携带环境标识。
- [风险] 调用方传入非预期字符串，可能导致查询结果偏差
  Mitigation：保持请求缺省按 `BS` 处理，并在接口文档中明确建议值集合和默认行为。

## Migration Plan

1. 在 `project` 模块新增名单管理接口（增删查白名单/黑名单）与放量接口，不替换旧接口。
2. 先维护好 Redis 名单，再在灰度环境以 `dry-run` 验证样本与统计口径（包括白名单/黑名单效果）。
3. 按 1% -> 3% -> 5% -> 10% -> ... -> 100% 执行，阶段间观察关键指标。
4. 如阶段异常，使用相同比例和反向 tag 执行回滚。
5. 全量完成后保留接口用于后续版本灰度发布复用。
