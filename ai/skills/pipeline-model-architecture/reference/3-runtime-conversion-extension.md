# 运行时、转换与扩展

## 运行时角色

模型定义的是“流水线怎么长”，执行链路负责“流水线怎么跑”。

常见业务流程里，模型会参与：

- 流水线创建
- 编辑保存
- 启动构建
- 构建详情展示
- 重试或重放
- YAML 与 Model 双向转换

如果你在排查执行顺序、状态推进、调度行为，通常应该继续看 `process-module-architecture`，而不是只停在模型层。

## YAML 与 Model 转换

模型和 YAML 转换时，常见风险点是：

- 字段语义并不完全一一对应
- 某些运行时字段不应进入 YAML
- 某些模板或扩展字段在 YAML 中表达方式不同
- 新增字段后，双向转换可能出现信息丢失

处理这类问题时：

1. 先定义字段归属：模型专有、YAML 专有，还是双向公共字段
2. 再确定向后兼容策略
3. 最后补转换、校验和默认值

YAML/PAC 相关细节，继续看 `yaml-pipeline-transfer`。

## 模型校验

模型校验的目标不是只保证“对象不为空”，而是保证：

- 结构合法
- 层级关系合法
- 特殊类型使用合法
- 运行前提可满足

高频校验点：

- 触发容器是否存在
- FinallyStage 是否位置正确
- Matrix 配置是否合法
- Element 类型和字段组合是否合法
- 必填字段、命名、依赖关系是否完整

## 扩展新类型

新增 `Element` 或 `Container` 类型时，至少检查下面几层：

1. 模型类定义
2. 多态序列化声明
3. 前端编排侧模型与表单
4. 校验逻辑
5. 转换逻辑
6. 执行链路识别逻辑

不要只加一个类就结束。模型扩展通常是跨层改动。

## 调试建议

- 先拿到真实 Model 或 YAML 样本，再判断是结构问题还是执行问题
- 优先比较“修改前 / 修改后 / 历史版本”三份数据
- 如果是老数据出错，先看兼容逻辑，不要直接改当前模型定义
- 如果是执行时出错，确认是不是 `Process` 或 YAML 转换层放大了模型问题

## 性能注意

模型过大、频繁序列化或频繁全量读取时，常见优化方向是：

- 降低无必要字段膨胀
- 避免不必要的全量转换
- 区分编辑态和执行态读取路径
- 把真正的热路径问题放回 `Process` 或查询层定位
