# 持久化与版本管理

## 持久化心智

流水线模型不是只存在于内存中的对象，它会以多种形态被保存和读取：

- 流水线当前版本配置
- 历史版本快照
- 构建时的模型快照
- 可能的 YAML 形态或模板实例化结果

所以修改模型字段时，必须同时考虑：

1. 新建流程是否可用
2. 老版本数据是否还能被读取
3. 构建重试或历史详情是否还能回放
4. 与模板、分支版本、YAML 转换是否兼容

## 常见持久化位置

排查模型落库时，通常会涉及：

- 流水线资源表
- 流水线信息表
- 版本表或版本状态相关表
- 构建详情 / 构建记录快照

如果问题更偏数据库设计而不是模型结构本身，切到 `database-design`。

## 读取链路

常见模型读取场景：

- 打开流水线编辑页
- 启动构建
- 查看构建详情
- 读取某个历史版本
- 模板实例化或模板回填

不要假设“所有读取都走同一个 Service”。很多看起来像模型问题，实际是读取链路走到了不同快照。

## 版本管理重点

模型版本常见关注点：

- 每次修改生成新版本
- 当前发布版本与历史版本的关系
- 分支版本或草稿版本的处理
- 历史版本是否允许重试、查看、回放

高风险改动：

- 删除旧字段但没有兼容读取逻辑
- 给新字段加了强依赖，却没有默认值
- 只处理“当前版本”，没处理历史版本或快照

## 序列化注意事项

模型广泛依赖 JSON 序列化和反序列化：

- 多态字段要保持可反序列化
- 默认值设计要稳定
- 废弃字段最好走兼容转换，而不是直接硬删
- 运行时字段和配置字段要区分清楚

排查序列化问题时，优先确认：

1. 类型声明是否还完整
2. 字段重命名是否有兼容
3. 历史 JSON 是否还能被当前代码解析
4. 前后端对同一字段语义是否一致

## 推荐排查顺序

1. 先确认问题发生在“当前配置”、“历史版本”还是“构建快照”
2. 再确认问题发生在“落库”、“读库”还是“序列化转换”
3. 最后再决定是改模型定义、兼容逻辑、Service 读取逻辑还是数据库结构
