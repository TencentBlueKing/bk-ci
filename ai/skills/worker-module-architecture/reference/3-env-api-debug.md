# 环境变量、API 与排查

## 环境变量的定位

- Worker 是变量传递的重要中继站
- 执行器会消费输入变量，也可能产出输出变量
- 变量问题既可能影响任务执行，也可能影响下游任务

## 变量排查顺序

1. 输入变量是否已进入 Worker
2. 执行过程中变量是否被覆盖或过滤
3. 输出变量是否按规则回传

## API 客户端关注点

- 日志、制品、流程等能力通常通过 Worker API 客户端访问后端
- 排查问题时要区分本地执行失败和远端接口失败
- 超时、重试、幂等和错误落盘要一起看

## 调试建议

- 先缩小到单个任务和最小变量集合
- 同时看本地执行结果、日志上报结果和接口返回
- 如果问题跨越 Agent、Worker、Dispatch，先确认断点发生在哪一层

## 容易混淆的边界

- 变量契约定义通常不在 Worker
- 调度与排队问题通常不在 Worker
- 插件字段设计通常不在 Worker
