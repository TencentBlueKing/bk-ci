# 插件基础与目录

## 先判断你在改什么

- 改插件定义、字段、入口命令、运行时：属于插件开发
- 改插件如何被 Worker 下载、执行、回传结果：属于 `worker-module-architecture`
- 改流水线里如何引用插件元素：通常还要看 `pipeline-model-architecture`

## 核心心智

- 流水线插件是流水线里的最小可复用执行单元
- 插件对外的核心契约是 `task.json`
- 代码语言只是实现方式，平台识别和渲染入口仍然围绕 `task.json`

## 开发时通常先定的三件事

1. 插件解决什么问题，输入输出是什么
2. 用什么语言和运行时
3. 目标入口、依赖命令和是否需要多平台配置

## 重点关注的配置块

- `atomCode`：插件唯一标识
- `execution`：运行语言、入口、依赖、运行时版本、OS 配置
- `input` / `inputGroups`：参数定义与展示
- `output`：输出变量契约
- `config`：插件控制特性

## 常见设计取舍

- 单平台插件更简单，但多平台插件更适合基础设施类能力
- Java 通常更适合复杂逻辑和长期维护
- Python / NodeJS 更适合快速实现和脚本类逻辑
- Golang 更适合分发单文件或平台相关能力

## 什么时候需要多平台配置

- 插件要在 Linux、Windows、macOS 上跑
- 各平台入口命令不同
- 二进制文件或依赖包因平台而异

## 容易踩坑的地方

- 只定义默认平台，忘了其他平台入口
- 把实验性字段提前暴露为正式输入
- 运行时版本不明确，后续兼容成本高
