# 调试、发布与最佳实践

## 调试时先区分问题层级

- 表单渲染不对：先查 `task.json`
- 输入值进入执行入口后异常：查插件实现
- 下载、执行、日志、产物上传异常：查 `worker-module-architecture`
- 构建机选择或排队异常：查 `dispatch-module-architecture`

## 调试重点

- 先固定一组最小输入，避免把问题复杂化
- 记录输入、关键中间状态、输出变量和错误码
- 避免把调试日志直接保留成正式输出或用户文案
- 本地能跑通不等于在 Worker 环境能跑通

## 发布前检查

- `task.json` 契约完整且字段命名稳定
- 敏感字段、默认值、必填规则已确认
- 运行时版本和目标平台已确认
- 输出变量、错误码、升级兼容策略已确认

## 最佳实践

- 优先做小而清晰的插件，不做大而全插件
- 输入字段先做“最少够用”，再逐步扩展
- 兼容性策略比一次性功能堆叠更重要
- 插件说明、错误码和输出变量要面向使用者，而不是面向作者自己

## 常见发布风险

- 插件版本发布后与旧流水线不兼容
- 不同平台入口不一致但没测试
- 本地依赖存在，Worker 环境不存在
- 错误码和日志无法帮助定位问题
