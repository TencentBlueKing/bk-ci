# 核心模式速查

## 工厂模式

适用问题：

- 根据类型、标识或配置创建不同实现
- 屏蔽调用方对具体实现类的感知
- 为插件、任务、代码库类型、处理器类型提供统一入口

在 BK-CI 中常见于：

- Task 创建
- SCM 实现选择
- 命令执行器选择
- 条件处理器创建

不要用在：

- 只有一个实现且没有扩展空间
- 只是为了少写一行 `new`

## 策略模式

适用问题：

- 一组规则可以互换
- 同一目标在不同条件下使用不同算法
- 需要把分支逻辑从主流程中抽离出去

在 BK-CI 中常见于：

- 权限校验策略
- 数据迁移策略
- 不同运行条件或不同渠道处理策略

不要用在：

- 分支极少且长期稳定
- 调用方并不真的需要切换策略

## 责任链模式

适用问题：

- 多个处理器按顺序处理同一请求
- 某一步可能终止，也可能继续传递
- 需要把大流程拆成多个独立节点

在 BK-CI 中常见于：

- 引擎控制链
- 构建后置处理
- 多步骤校验或过滤链

不要用在：

- 只有单步逻辑
- 顺序固定但每一步高度耦合，拆开反而更难维护

## 观察者模式

适用问题：

- 某个核心事件发生后，要通知多个下游处理器
- 希望生产者和消费者解耦
- 适合事件总线、Spring Event、监听器体系

在 BK-CI 中常见于：

- Spring Event 监听
- 构建状态变化后的通知或后置动作

不要用在：

- 强一致、强事务主链路
- 调用方必须立即拿到确定结果的同步流程

## 模板方法模式

适用问题：

- 主流程稳定
- 某几个步骤需要按实现差异覆盖
- 希望子类复用框架而只改钩子点

在 BK-CI 中常见于：

- 后置处理器
- 某类统一执行流程下的差异化实现

不要用在：

- 主流程本身也经常变化
- 差异点太多，继承层次会迅速失控

## 装饰器模式

适用问题：

- 在不改核心对象的情况下附加行为
- 适合日志、权限、异常包装、能力增强

在 BK-CI 中常见于：

- 权限服务增强
- 异常或结果包装

不要用在：

- 多层叠加后让调用链难以理解
- 需要的是流程控制而不是行为增强
