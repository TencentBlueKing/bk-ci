# 模式选型指南

## 先问 4 个问题

在决定是否引入设计模式前，先回答下面 4 个问题：

1. 变化点到底在哪里？
2. 这种变化是短期一次性，还是会持续扩展？
3. 调用方需要“创建对象”“切换算法”“串联处理”，还是“监听事件”？
4. 项目里有没有已有同类实践可以直接复用？

## 选择路径

### 场景 1：我需要按类型创建不同实现

优先考虑：工厂模式

典型信号：

- 输入是 `type`、`classType`、code、枚举
- 输出是一组实现中的某一个
- 调用方不想感知具体类名

### 场景 2：我需要在多个算法中切换一种

优先考虑：策略模式

典型信号：

- 行为目标相同，规则不同
- 不同条件对应不同处理方式
- 主流程不想继续堆条件分支

### 场景 3：我要把一大段处理流程拆成多个节点

优先考虑：责任链模式

典型信号：

- 每一步都是可独立理解的处理器
- 顺序明确
- 某一步可能拦截，也可能继续

### 场景 4：一个核心动作发生后，要通知多个下游

优先考虑：观察者模式

典型信号：

- 下游处理器数量会变
- 生产者不应感知所有消费者
- 适合异步或事件驱动

### 场景 5：流程骨架固定，只允许局部覆写

优先考虑：模板方法模式

典型信号：

- 步骤顺序稳定
- 差异点有限且位置固定
- 继承比组合更自然

### 场景 6：我要给现有能力叠加附加行为

优先考虑：装饰器模式

典型信号：

- 核心对象已经存在
- 想增强而不是改写它
- 增强逻辑可组合、可拆卸

## 什么时候不要上模式

- 分支极少且不会扩展
- 团队已经看不懂现有抽象，再加一层只会更糟
- 这个问题更应该通过模块拆分、接口收敛或数据结构优化解决
- 你只是想让代码“看起来高级”

## BK-CI 特别注意

- 模式选型不能破坏现有模块边界
- 优先复用已有框架和项目模式，而不是自创抽象
- 如果已经有通用 skill 覆盖某种模式实践，优先链接它，不重复写一套百科
