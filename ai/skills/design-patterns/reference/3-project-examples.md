# BK-CI 项目案例

## 工厂模式

典型落点：

- `TaskFactory`
- `ScmFactory`
- `ApiFactory`
- `CommandFactory`

适合原因：

- 输入通常是插件类型、代码库类型、命令类型
- 调用方只关心“拿到正确实现”，不关心具体构造细节

## 策略模式

典型落点：

- 权限检查策略
- 数据迁移策略
- 某些按渠道或类型分支的处理策略

适合原因：

- 同一业务目标下存在多个可替换规则
- 把规则从主流程中抽离后，可读性明显提升

## 责任链模式

典型落点：

- 引擎控制链
- 多阶段后置处理器
- 某些过滤和校验链

适合原因：

- 整体流程天然可拆成多个步骤
- 每个节点职责清晰，便于插拔和排序

## 观察者模式

典型落点：

- Spring Event 监听器
- 事件发布后的通知、统计、后置动作

适合原因：

- 核心触发方不需要知道所有消费者
- 新增监听器时对主流程影响较小

## 模板方法模式

典型落点：

- 统一处理框架下的后置处理器
- 固定主流程、局部差异化的执行器

适合原因：

- 主流程相对稳定
- 差异点可以收敛到少量钩子方法

## 装饰器模式

典型落点：

- 权限服务增强
- 异常结果包装

适合原因：

- 希望在不改变核心能力的前提下附加责任
- 增强逻辑可被移除或组合

## 怎么使用这些案例

1. 先把你的问题归到“创建 / 切换 / 串联 / 监听 / 增强”哪一类
2. 再找项目中最接近的现有案例
3. 先复用已有结构，再决定是否要引入新的抽象

如果你的问题已经明显落在某个具体模块里，接下来优先切到对应模块 skill，而不是继续停留在本 skill。
