# 资源类型与操作设计

## 先定义的不是代码，而是模型

- 资源类型 ID 是全局契约
- 操作列表决定授权粒度
- 父子资源关系决定实例选择和申请体验

## 设计时先回答四个问题

1. 新资源在业务上归属于谁
2. 创建前资源是否存在
3. 用户通常会对它做哪些动作
4. 哪些动作是高频，哪些动作应被管理权限兜住

## 操作设计心智

- `create` 常常关联父资源，而不是新资源自身
- `list` / `view` / `edit` / `delete` / `manage` 要区分好是否实例级
- 不要为了“看起来完整”机械复制所有动作

## 常见坑

- 资源 ID、枚举值、IAM 配置 ID 命名不一致
- 父资源关系设计错，导致申请链路异常
- 动作粒度过粗或过细，后续难维护
