# PAC 与模板

## PAC 是什么

在这个 skill 里，PAC 关注的是：

- YAML 存在代码库中
- 蓝盾从代码库读取 YAML
- 经过模板、表达式、切面等处理后转换成可执行模型

PAC 不只是“从 Git 读一份 YAML”，而是“代码库内容如何进入流水线体系”的完整转换链。

## PAC 常见问题

高频场景：

- 代码库里的 YAML 能否正确被识别
- PAC 同步后，模板是否正确展开
- 分支、路径、触发器、变量是否正确注入
- PAC 和普通手工编辑流水线之间的差异如何处理

如果问题偏代码库接入或 Webhook，本质更靠近 `repository-module-architecture`。

## 模板体系

YAML 模板常见分为：

- `extends` 模板引用
- Step 模板
- Job 模板
- Stage 模板

这些模板能力会直接影响：

- YAML 结构展开
- 默认值填充
- 参数覆盖
- 导入导出时的稳定性

## 模板排查顺序

1. 先确认模板引用关系是否解析成功
2. 再确认模板参数是否传递正确
3. 再确认展开后结果是否符合 `Model` 结构预期
4. 最后检查导出时是否还能保留合理的模板表达

## PAC 与模板的交叉问题

PAC 和模板经常一起出问题，典型原因：

- 代码库中的 YAML 依赖模板，但模板上下文没有准备好
- 模板展开前后，表达式或变量语义变化
- 代码库分支、路径、权限或缓存导致拿到的不是预期 YAML

## 建议

- 对 PAC 场景，优先保留“代码库原始 YAML -> 展开后 YAML / Model”的对照样本
- 模板改动后，不只验证单个模板，还要验证被它引用的真实流水线
- 不要把 PAC 问题直接归因为 YAML 语法错误，很多问题发生在模板或上下文准备阶段
