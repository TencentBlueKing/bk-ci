# Ask、心跳与任务拉起

## Ask 模式的定位

- Agent 通过统一 Ask 请求和服务端交互
- 一次交互里可能同时包含心跳、升级、构建和其他任务信息
- 这是理解 Agent 主循环的关键入口

## 主循环心智

- 启动时先初始化配置和环境
- 持续执行 Ask 轮询
- 根据响应内容分别处理心跳、构建、升级和 Pipeline 任务

## 看问题时先分层

- 心跳不正常：先看心跳组装、发送和响应处理
- 接不到任务：先看 Ask 请求能力位和服务端响应
- 拉起 Worker 失败：再看本地进程、环境和文件

## 常见坑

- 把服务端没下发任务误判成 Agent 没请求到
- 心跳成功但构建能力位没打开
- Worker 进程问题被误判成 Ask 逻辑问题
