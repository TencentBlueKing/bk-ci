# 模板基础与版本体系

## 模板的核心价值

模板模块解决的是“把一条或一组流水线定义抽象成可复用资产”，常见价值包括：

- 配置复用
- 标准化落地
- 版本控制
- 批量实例化
- 跨项目或跨组织共享

## V1 与 V2

模板体系通常会同时存在 V1 和 V2 两套实现。

理解方式：

- V1：更偏历史兼容与基础功能
- V2：更偏完善的版本体系、草稿、复杂场景、PAC 等新能力

排查模板问题时，先确认请求入口、Service、DAO 和表结构到底落在哪一套。

## 模板类型

常见模板类型通常包括：

- 项目内自定义模板
- 约束模板 / 商店来源模板
- 公共模板

这些类型往往直接影响：

- 创建入口
- 权限边界
- 是否可发布或来自商店
- 版本变更规则

## 目录与分层心智

模板模块常见会覆盖：

- API 接口层
- POJO 数据模型
- 业务服务层
- 数据访问层
- 权限实现

如果你只是想找代码，不要停在本 skill，直接去相关模块查 Resource / Service / Dao。

## 数据与资源

模板相关数据通常至少包括：

- 模板基础信息
- 模板资源内容
- 模板与流水线关联
- 模板实例关系
- 版本状态或迁移记录

不要把“模板信息表”和“模板资源表”混为一谈。很多版本问题不是信息错，而是资源版本错。

## 什么时候要切到别的 skill

- 改模型结构：`pipeline-model-architecture`
- 改 YAML / PAC 转换：`yaml-pipeline-transfer`
- 改权限：`auth-module-architecture`
- 改商店分发：`store-module-architecture`
