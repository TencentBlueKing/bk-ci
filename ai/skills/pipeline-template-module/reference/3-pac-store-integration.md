# PAC 模板与商店集成

## PAC 模板

PAC 模板不是单纯“模板 + YAML”，而是下面几件事的组合：

- 模板来自或作用于 YAML 流水线
- 模板版本和 YAML 转换要保持一致
- 代码库上下文、分支、路径、Webhook 等因素会影响模板结果

因此 PAC 模板问题通常同时涉及：

- 模板模块
- YAML 转换模块
- 代码库接入

## PAC 模板常见问题

- 模板能创建，但 YAML 转换失败
- 代码库里的模板内容和平台侧模板版本不一致
- PAC 分支版本与模板版本映射混乱
- 模板展开后结果不符合预期

如果更像代码库触发、Webhook 或仓库路径问题，继续看 `repository-module-architecture`。

## 商店模板

商店模板与项目内模板的关键差异通常在于：

- 来源不同
- 权限边界不同
- 安装与升级链路不同
- 版本和元数据管理更强调分发

所以不要用项目内模板的心智直接套商店模板。

## 模板安装与导入

商店模板常见链路包括：

1. 从商店拉取模板元数据
2. 安装到项目或平台内
3. 生成本地可用模板记录
4. 后续升级、同步或迁移

高风险点：

- 安装成功但资源内容不完整
- 安装版本和本地版本映射不清
- 升级时覆盖了本地定制

## 建议

- PAC 模板问题要保留“原始 YAML / 展开后结果 / 模板版本 / 最终模型”四份对照
- 商店模板问题要同时看来源版本、本地安装版本和关联实例
- 模板来源一旦变化，不要只验证创建流程，还要验证升级与回滚
