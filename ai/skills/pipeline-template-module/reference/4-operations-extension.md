# 排查、事件、锁与扩展

## 事件驱动

模板模块里常见会有事件驱动机制，例如：

- 模板实例化后的后置处理
- 版本变更后的通知或同步
- 某些批量操作后的异步收敛

如果问题表现为“主流程已成功，但后续状态不一致”，要优先怀疑事件监听或异步处理。

## 分布式锁

模板相关锁通常出现在：

- 模板发布
- 批量实例化
- 版本切换
- 商店安装或迁移

锁问题的典型表现：

- 并发下出现重复版本
- 批量操作部分覆盖
- 状态看起来不一致

不要看到“偶现失败”就只查接口层，很多模板问题是并发保护没生效。

## OpenAPI 与外部入口

模板模块可能会暴露：

- 用户接口
- 服务间接口
- 跨项目接口
- OpenAPI 或运营管理接口

排查时先区分入口，否则很容易误判为什么同样的模板动作在不同接口行为不同。

## 扩展模板能力

新增模板相关功能时，至少要检查：

1. 模板数据模型
2. 模板资源持久化
3. 版本生成与校验
4. 实例化链路
5. 权限控制
6. 如果涉及 YAML，还要检查 `yaml-pipeline-transfer`

## 常见排查路径

### 问题 1：模板创建成功，但实例化异常

优先检查：

- 模板资源是否完整
- 参数映射是否正确
- 实例化入口是否匹配当前模板版本

### 问题 2：版本切换后历史实例异常

优先检查：

- 模板版本和实例版本关系
- 草稿 / 正式 / 分支版本状态
- 兼容逻辑是否覆盖旧数据

### 问题 3：PAC 模板导入或发布异常

优先检查：

- YAML 转换链
- 模板展开逻辑
- 代码库上下文是否正确

## 性能注意

模板问题一旦涉及批量实例化、批量升级或商店安装，容易放大为性能问题。排查时优先区分：

- 是模板读写慢
- 是实例化链路慢
- 是异步后处理慢
- 还是权限 / 锁争用导致耗时上升
