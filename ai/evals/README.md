# BK-CI AI Harness Evals

`ai/evals/` 用来沉淀真实 BK-CI AI 编码任务，验证 rules、skills 和编辑器适配是否有效。

评测任务不是通用 benchmark，而是团队自己的回放样本。它应该回答一个问题：在当前 Harness 下，AI
是否能按 BK-CI 的模块边界、文档流程和代码规范完成任务。

## 任务选择原则

优先选择这些任务：

- 真实发生过，且人工可以判断好坏
- AI 容易误判模块边界或遗漏前置 skill
- 需要遵守 OpenSpec、Kotlin、Spring Boot、前端或测试约定
- 有明确验证信号，例如单测、lint、接口契约或人工验收点
- 返工成本较高，值得沉淀为回放样本

暂不选择这些任务：

- 只改一两个错别字
- 完全依赖私有外部系统且无法复现
- 验收标准只能靠主观偏好判断
- 已经过时、不会再出现的迁移任务

## 推荐格式

每个任务建议单独一个 Markdown 文件，命名为：

```text
YYYYMMDD-short-task-name.md
```

文件内容建议包含：

```markdown
# 任务名称

## 背景

说明任务来自哪里、为什么有代表性。

## 用户请求

保留可复用的原始请求，必要时脱敏。

## 期望 Harness 行为

- 应先读取哪些 rule 或 skill
- 是否需要进入 Plan Mode
- 是否需要先更新 OpenSpec
- 允许修改哪些文件范围

## 验证信号

- 需要运行的检查
- 需要人工确认的点
- 明确不应发生的行为

## 评测记录

- 编辑器：
- 模型：
- 结果：
- 返工原因：
- 应反哺到 rule / skill / reference / tests 的建议：
```

## 评测指标

首批先用轻量指标，不急于平台化：

- 是否读对 skill
- 是否遵守 OpenSpec 优先规则
- 是否把修改限制在合理范围
- 是否通过必要验证
- 是否产生无关重构或上下文污染
- 结果是否被保留，或需要人工大幅返工

## 使用方式

1. 从真实任务中挑选 5 到 10 个样本。
2. 用 Cursor、CodeBuddy 或不同模型分别执行同一任务。
3. 记录结果和失败归因。
4. 只把重复出现的问题反哺到 rule 或 skill。
5. 每次 Harness 调整后，回放一小批任务验证收益。
